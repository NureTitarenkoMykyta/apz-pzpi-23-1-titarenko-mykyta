3	ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про систему
Назва системи: Netflix.
Розробник: Netflix, Inc.
Область застосування: Стрімінговий мультимедійний сервіс для надання послуг відео за запитом (Video on Demand) у глобальному масштабі.
Користувачі: Понад 260 мільйонів активних передплатників по всьому світу.
Основні функції: Пошук та відтворення відеоконтенту, персоналізовані рекомендації, підтримка декількох профілів користувачів, адаптивний стрімінг (підлаштування якості відео під швидкість інтернету), офлайн-перегляд та багатоплатформена підтримка.
3.2 Архітектурний стиль системи
Основним архітектурним стилем Netflix є хмарна мікросервісна архітектура, яка поєднує в собі елементи подієво-орієнтованого підходу (Event-Driven) та багатошаровості.
Вибір зазначеного архітектурного стилю зумовлений стратегічними вимогами до масштабованості, доступності та швидкості розробки:
Горизонтальна масштабованість та еластичність: На відміну від монолітних систем, мікросервісна архітектура дозволяє масштабувати окремі функціональні блоки незалежно один від одного. Використання хмарної інфраструктури (AWS) забезпечує динамічне виділення ресурсів: у періоди пікових навантажень система автоматично розгортає додаткові екземпляри критично важливих сервісів, мінімізуючи витрати в періоди низької активності.
Висока відмовостійкість (Resilience): Архітектура спроєктована з урахуванням принципу ізоляції збоїв. Завдяки механізмам «Circuit Breaker» та автономності сервісів, несправність одного компонента (наприклад, системи відгуків) не призводить до каскадного падіння всієї платформи. Це гарантує безперебійність основної функції сервісу — відтворення контенту.
Асинхронність та цілісність даних: Подієво-орієнтований підхід, реалізований через шину даних Apache Kafka, дозволяє системі опрацьовувати колосальні обсяги інформації без блокування основних потоків. Це забезпечує актуальність рекомендаційних алгоритмів та аналітичних звітів у режимі «eventual consistency» (кінцевої узгодженості), що є критичним для глобально розподіленої системи.
3.3 Основні компоненти системи
Інтерфейс користувача: клієнтські додатки для різних платформ: Smart TV, мобільні пристрої (iOS/Android), веб-браузери та ігрові консолі. Основне завдання — забезпечення взаємодії з користувачем та візуалізація контенту. Велика увага приділяється збору даних про взаємодію (кліки, час перегляду), які потім передаються на сервер для аналітики.
Серверна частина (Control Plane): функціонує переважно в хмарі AWS. Вона координує всі запити від додатків. Тут працює API Gateway (Zuul), який приймає вхідні запити, перевіряє аутентифікацію та направляє їх до відповідних внутрішніх мікросервісів.
Рівень бізнес-логіки (Application Layer): складається з тисяч мікросервісів. Кожен із них виконує окрему бізнес-задачу:
—	Сервіс підписок: Керує платіжними циклами та доступами.
—	Система рекомендацій: На основі алгоритмів машинного навчання формує персоналізовані списки фільмів.
—	Сервіс відтворення: Визначає, який саме файл (якість, кодек) найкраще підходить для конкретного пристрою користувача.
Рівень зберігання даних (Data Layer):
—	Netflix використовує стратегію Polyglot Persistence, обираючи базу даних під конкретну задачу:
—	Cassandra: Для зберігання великих обсягів структурованих даних (історія переглядів).
—	MySQL: Для критичних транзакцій та даних білінгу.
—	EVCache: Швидке сховище в оперативній пам'яті для кешування результатів запитів, що часто повторюються.
—	S3 (Amazon Simple Storage Service): Для зберігання вихідних медіафайлів високої якості.
Допоміжні сервіси:
—	Open Connect (CDN): Мережа серверів по всьому світу, на яких зберігаються копії відеофайлів для швидкої доставки користувачам.
—	Транскодування: Система автоматичного перетворення відео у сотні різних форматів і роздільних здатностей відразу після завантаження оригіналу.
—	Chaos Monkey: Інструмент для автоматичного тестування стійкості системи шляхом випадкового вимкнення окремих компонентів у робочому середовищі.
3.4 Взаємодія компонентів
Потоки даних (Data Flows):
Потік запитів користувача (Control Plane):
—	Клієнтський додаток надсилає запит на API Gateway (Zuul).
—	Zuul виконує автентифікацію та перенаправляє запит до відповідного мікросервісу.
—	Мікросервіс звертається до бази даних або кешу (EVCache) для отримання інформації (наприклад, списку фільмів).
—	Дані повертаються користувачу у форматі JSON.
Потік медіаданих (Data Plane):
—	Після вибору фільму клієнт звертається до сервісу відтворення для отримання URL-адреси відеопотоку.
—	Клієнт з'єднується з найближчим сервером мережі Open Connect (CDN).
—	Відео передається частинами (чанками), при цьому клієнт постійно оцінює швидкість інтернету та динамічно перемикається між файлами різної якості (Adaptive Bitrate Streaming).

Протоколи взаємодії:
HTTP/2 та REST: Основні протоколи для взаємодії між клієнтськими датками та API Gateway. 
gRPC: Використовується для високопродуктивного зв'язку між утрішніми мікросервісами. 
TLS (Transport Layer Security): Забезпечує шифрування всіх даних, що едаються між клієнтом і хмарою. 
TCP/UDP: Використовуються для безпосередньої передачі відеопотоку з CDN-серверів до пристрою користувача.
Механізми синхронізації та обміну даними:
— Асинхронний обмін через Apache Kafka: Більшість внутрішніх процесів не чекають миттєвої відповіді; подія записується в чергу, а інші сервіси опрацьовують її за наявності ресурсів. 
— Eventual Consistency (Кінцева узгодженість): Дані в різних географічних регіонах синхронізуються з невеликою затримкою, що забезпечує стабільну роботу системи навіть при розриві зв'язку між дата-центрами. 
— Кешування (EVCache): Результати складних запитів зберігаються в оперативній пам'яті для миттєвого доступу, що зменшує навантаження на основні бази даних.
3.5 Масштабованість та надійність
Механізми балансування навантаження:
— Elastic Load Balancing (ELB): Сервіс від AWS, який першим приймає запити з інтернету та розподіляє їх між шлюзами Zuul.
— Zuul (API Gateway): Виконує функцію інтелектуального балансувальника на рівні додатку, направляючи запити на конкретні екземпляри мікросервісів залежно від їхньої завантаженості.
— Client-side Load Balancing (Ribbon): Дозволяє одному мікросервісу самостійно обирати, до якого саме екземпляра іншого сервісу звернутися, орієнтуючись на дані про їхню доступність.
Відмовостійкість:
— Hystrix (Circuit Breaker): Механізм, який «розмикає ланцюг», якщо певний мікросервіс починає працювати з помилками. Це дозволяє системі не чекати на відповідь від поламаного компонента, а відразу видавати стандартну відповідь або порожній результат.
— Chaos Engineering: Використання інструментів (Chaos Monkey), які навмисно спричиняють збої в реальному часі для перевірки здатності системи до самовідновлення.
— Регіональна ізоляція: Netflix працює у трьох незалежних регіонах AWS. Якщо один регіон повністю виходить із ладу, весь трафік автоматично перенаправляється на два інших.
Кешування:
— EVCache: Глобально розподілене сховище в оперативній пам'яті (на базі Memcached), де зберігаються профілі користувачів, персоналізовані підбірки та метадані фільмів. 
— Локальне кешування: Клієнтські додатки зберігають частину статичних даних (зображення обкладинок, тексти описів) безпосередньо в пам'яті пристрою.
Реплікація:
— Cross-Region Replication: Усі дані в базі Cassandra автоматично копіюються між різними географічними регіонами. Це гарантує, що ваші налаштування будуть однаковими, де б ви не авторизувалися.
— Open Connect Replication: Новий контент (фільми та серіали) реплікується на тисячі CDN-серверів по всьому світу в години найменшого навантаження на мережу (вночі).
Хмарна інфраструктура та контейнеризація:
— AWS (Amazon Web Services): Забезпечує практично безмежну обчислювальну потужність, дозволяючи Netflix автоматично додавати тисячі серверів за лічені хвилини при зростанні трафіку.
— Titus: Власна система управління контейнерами, яка дозволяє запускати тисячі мікросервісів у ізольованих середовищах. Це спрощує масштабування та забезпечує ідентичність роботи коду на комп'ютері розробника і в реальній системі.
3.6 Приклади програмної реалізації
SimplePreFilter (API Gateway Pattern):
Цей фрагмент коду реалізує функціональність Zuul Filter. У мікросервісній архітектурі шлюз виступає єдиною точкою входу, де цей фільтр виконує роль «Pre-filter» — обробника запиту до його передачі безпосередньо в сервіс. Він дозволяє централізовано керувати безпекою, автентифікацією та трасуванням запитів, додаючи унікальні ідентифікатори (Trace ID) для моніторингу шляху запиту крізь десятки різних сервісів.
Circuit Breaker (Hystrix Command):
Фрагмент демонструє механізм відмовостійкості. Патерн «автоматичного вимикача» (Circuit Breaker) захищає систему від каскадних збоїв. Якщо зовнішній сервіс (наприклад, рекомендацій) не відповідає або працює занадто повільно, Hystrix перериває зв'язок і виконує метод getFallback. Це дозволяє користувачу отримати альтернативний результат (наприклад, список популярних фільмів) замість повідомлення про критичну помилку всієї платформи.
Feign Client (Service Discovery & Communication):
Цей інтерфейс ілюструє декларативний підхід до взаємодії між мікросервісами. Замість написання складного коду для обробки HTTP-запитів, розробник просто описує контракт. Завдяки інтеграції з сервісом виявлення (Eureka), Feign автоматично дізнається актуальну IP-адресу потрібного мікросервісу та забезпечує надійний зв'язок між компонентами системи, роблячи мережеву взаємодію прозорою для бізнес-логіки.

Запит:
Протоколи взаємодії, які використовує сервіс Netflix?
Відповідь:
Для забезпечення стабільної роботи за різних умов мережі та на різних пристроях, Netflix використовує стек протоколів, кожен з яких оптимізовано під конкретну архітектурну задачу.
1. HTTP/2 та REST
Це основний протокол для зовнішньої взаємодії між клієнтськими пристроями (телевізори, смартфони) та API-шлюзом.
— REST (Representational State Transfer): Використовується для стандартних операцій, таких як завантаження метаданих фільмів, управління профілями та пошук.
— HTTP/2: Завдяки мультиплексуванню (передачі кількох запитів через одне з'єднання) цей протокол значно прискорює завантаження інтерфейсу користувача.
2. gRPC (Remote Procedure Call)
Використовується переважно для внутрішньої взаємодії між тисячами мікросервісів у хмарі AWS.
— gRPC працює поверх HTTP/2 та використовує Protocol Buffers (двійковий формат даних).
— Це забезпечує набагато вищу швидкість передачі та менший обсяг даних порівняно з текстовим JSON, що критично важливо при мільйонах міжсервісних запитів на секунду.
3. TLS (Transport Layer Security)
Весь трафік Netflix (як дані користувача, так і відеопотоки) обов'язково шифрується за допомогою TLS 1.2 або 1.3.
— Це захищає конфіденційність користувачів та запобігає перехопленню даних або підміні контенту зловмисниками.
4. Адаптивні протоколи стрімінгу (DASH та HLS)
Для безпосередньої передачі відео Netflix використовує протоколи на базі сегментованого HTTP:
— MPEG-DASH (Dynamic Adaptive Streaming over HTTP): Основний стандарт для більшості сучасних пристроїв.
— HLS (HTTP Live Streaming): Використовується переважно для пристроїв Apple.
— Ці протоколи дозволяють плеєру динамічно змінювати якість відео (бітрейт) залежно від поточної швидкості інтернету, уникаючи зупинок на буферизацію.
Запит: 
Механізми балансування навантаження, які використовує сервіс Netflix?
Відповідь:
Для забезпечення стабільної роботи при величезних обсягах трафіку Netflix використовує багаторівневу стратегію балансування навантаження. Вона охоплює шлях від глобального DNS-запиту до вибору конкретного екземпляра мікросервісу всередині хмари.
Нижче наведено ключові механізми:
1. Глобальне балансування (GSLB)
На найвищому рівні Netflix використовує Global Server Load Balancing на базі DNS та Route53 від AWS.
— Коли ви відкриваєте додаток, система визначає вашу геопозицію та спрямовує запит до найближчого регіону AWS (наприклад, US-East чи EU-West).
— Це дозволяє рівномірно розподілити навантаження між дата-центрами в різних частинах світу та мінімізувати затримки (latency).
2. Зовнішнє балансування (AWS ELB)
Після того, як запит потрапив у конкретний регіон, він проходить через Elastic Load Balancing (ELB) від Amazon.
— Це апаратний/програмний рівень, який розподіляє вхідний HTTP-трафік між масивом шлюзів Zuul.
— Його головне завдання — переконатися, що жоден сервер шлюзу не перевантажений, і автоматично виключати з ротації вузли, які стали непрацездатними.
3. Динамічний шлюз (Zuul API Gateway)
Zuul — це власна розробка Netflix, яка виконує роль інтелектуального балансувальника на рівні прикладного інтерфейсу.
— Він не просто пересилає запити, а аналізує їх у реальному часі.
— Zuul може виконувати "канаркові релізи" (спрямовувати 1% трафіку на нову версію сервісу для тестування) або відхиляти запити від конкретних пристроїв, якщо вони генерують аномальне навантаження.
4. Балансування на стороні клієнта (Ribbon)
Це один із найбільш унікальних механізмів Netflix. Замість того, щоб покладатися на центральний сервер-балансувальник всередині мережі, кожен мікросервіс сам вирішує, куди відправити запит.
— Ribbon — це бібліотека, інтегрована в кожен мікросервіс.
— Вона отримує список доступних адрес з реєстру Eureka і самостійно обирає найкращий вузол для звернення.
— Це прибирає "зайвий крок" (hop) через центральний балансувальник, зменшуючи затримки та усуваючи єдину точку відмови.
