Запит:
Протоколи взаємодії, які використовує сервіс Netflix?
Відповідь:
Для забезпечення стабільної роботи за різних умов мережі та на різних пристроях, Netflix використовує стек протоколів, кожен з яких оптимізовано під конкретну архітектурну задачу.
1. HTTP/2 та REST
Це основний протокол для зовнішньої взаємодії між клієнтськими пристроями (телевізори, смартфони) та API-шлюзом.
— REST (Representational State Transfer): Використовується для стандартних операцій, таких як завантаження метаданих фільмів, управління профілями та пошук.
— HTTP/2: Завдяки мультиплексуванню (передачі кількох запитів через одне з'єднання) цей протокол значно прискорює завантаження інтерфейсу користувача.
2. gRPC (Remote Procedure Call)
Використовується переважно для внутрішньої взаємодії між тисячами мікросервісів у хмарі AWS.
— gRPC працює поверх HTTP/2 та використовує Protocol Buffers (двійковий формат даних).
— Це забезпечує набагато вищу швидкість передачі та менший обсяг даних порівняно з текстовим JSON, що критично важливо при мільйонах міжсервісних запитів на секунду.
3. TLS (Transport Layer Security)
Весь трафік Netflix (як дані користувача, так і відеопотоки) обов'язково шифрується за допомогою TLS 1.2 або 1.3.
— Це захищає конфіденційність користувачів та запобігає перехопленню даних або підміні контенту зловмисниками.
4. Адаптивні протоколи стрімінгу (DASH та HLS)
Для безпосередньої передачі відео Netflix використовує протоколи на базі сегментованого HTTP:
— MPEG-DASH (Dynamic Adaptive Streaming over HTTP): Основний стандарт для більшості сучасних пристроїв.
— HLS (HTTP Live Streaming): Використовується переважно для пристроїв Apple.
— Ці протоколи дозволяють плеєру динамічно змінювати якість відео (бітрейт) залежно від поточної швидкості інтернету, уникаючи зупинок на буферизацію.
Запит: 
Механізми балансування навантаження, які використовує сервіс Netflix?
Відповідь:
Для забезпечення стабільної роботи при величезних обсягах трафіку Netflix використовує багаторівневу стратегію балансування навантаження. Вона охоплює шлях від глобального DNS-запиту до вибору конкретного екземпляра мікросервісу всередині хмари.
Нижче наведено ключові механізми:
1. Глобальне балансування (GSLB)
На найвищому рівні Netflix використовує Global Server Load Balancing на базі DNS та Route53 від AWS.
— Коли ви відкриваєте додаток, система визначає вашу геопозицію та спрямовує запит до найближчого регіону AWS (наприклад, US-East чи EU-West).
— Це дозволяє рівномірно розподілити навантаження між дата-центрами в різних частинах світу та мінімізувати затримки (latency).
2. Зовнішнє балансування (AWS ELB)
Після того, як запит потрапив у конкретний регіон, він проходить через Elastic Load Balancing (ELB) від Amazon.
— Це апаратний/програмний рівень, який розподіляє вхідний HTTP-трафік між масивом шлюзів Zuul.
— Його головне завдання — переконатися, що жоден сервер шлюзу не перевантажений, і автоматично виключати з ротації вузли, які стали непрацездатними.
3. Динамічний шлюз (Zuul API Gateway)
Zuul — це власна розробка Netflix, яка виконує роль інтелектуального балансувальника на рівні прикладного інтерфейсу.
— Він не просто пересилає запити, а аналізує їх у реальному часі.
— Zuul може виконувати "канаркові релізи" (спрямовувати 1% трафіку на нову версію сервісу для тестування) або відхиляти запити від конкретних пристроїв, якщо вони генерують аномальне навантаження.
4. Балансування на стороні клієнта (Ribbon)
Це один із найбільш унікальних механізмів Netflix. Замість того, щоб покладатися на центральний сервер-балансувальник всередині мережі, кожен мікросервіс сам вирішує, куди відправити запит.
— Ribbon — це бібліотека, інтегрована в кожен мікросервіс.
— Вона отримує список доступних адрес з реєстру Eureka і самостійно обирає найкращий вузол для звернення.
— Це прибирає "зайвий крок" (hop) через центральний балансувальник, зменшуючи затримки та усуваючи єдину точку відмови.
